# fm-os — the hub router

`/FM-os` is the single entry point to everything the hub carries. It does **not** inline every
tool — it keeps a small always-on **backbone** and reveals each toolset **on demand**. Same design
as a well-run agent: think with the backbone, act with an attachment loaded only when its trigger
fires.

## When to use (trigger)

Invoke when the user says "/FM-os", "use FM-os", "which FM-os tool for…", or asks for anything in
foundation-model / SLM ops — pretraining, fine-tuning, RL/post-training, serving, evaluation,
retrieval, curation, or working from one of the cited repos.

## Backbone (always on)

The FM-os discipline governs every route:

- **Loops, not one-shots** — organize work as gated turns that compound.
- **No evidence ⇒ no claim** — a result isn't real until a verifier/eval says so; report the null.
- **Verify at the artifact** — trust the run, the file, the test — not the label.
- **Spec-as-data** — the toolset index below is *generated* from `data/registry.yml` +
  `distill/manifest.json`, so it can never drift from what actually exists.
- **Maker ≠ checker** — the thing that generates is never the thing that certifies.

## How to route (progressive disclosure)

1. Match the request to ONE row in the index below.
2. **Read that row's `load` path** (`<path>/SKILL.md`) and follow it — pull in only that tool.
3. Don't carry more than one tool's body at a time; backbone bloat is a real cost.
4. Prefer a **certified** curated skill; a **distilled** scaffold is a starting point (read the
   upstream repo before running its code — untrusted until reviewed).
5. No matching tool? Say so, and point at the closest repo/paper in the README — don't fake a fit.

## Safety

Curated skills are certified (security-gated). Distilled scaffolds wrap **upstream, untrusted**
code — read it before you run it, prefer ephemeral use, human-gate anything that executes.

## Cross-runtime

One `SKILL.md`; the router index is plain markdown any harness (Claude Code / Codex / Hermes) can
read and act on.
